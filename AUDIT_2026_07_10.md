# OPAL-OKB: Комплексный архитектурный аудит
## 10 июля 2026

Анализ проведён по 4 направлениям: архитектура, качество кода, физика/алгоритмы, тесты/CI.

## 🔴 Критические проблемы (8)

### Физика
1. **Производная асферического полинома неверна** (`ray_tracing.py:~75,~165`)
   - `A₄→3r²` вместо `4r³`, `A₆→5r⁴` вместо `6r⁵`
   - Неправильные AND коэффициент, AND степень r
   - Влияние: все асферические поверхности дают неверные результаты

### Архитектура
2. **`analysis_gui.py` — монолит 4022 строки** (24 класса виджетов + оркестратор)
3. **`MainWindow` — God Object ~1070 строк** (~15 зон ответственности)
4. **Тройной путь результатов анализа** — 3 разных формата dict keys, 3 места обновления
5. **Нет пакетной структуры** — 20+ `.py` файлов в корне, `sys.path.insert` хаки

### Качество кода
6. **Создание лучей дублируется 15+ раз** — одинаковый паттерн `Ray(z=-50, k=sin...)` по всему коду
7. **Декодер LBO — одна функция ~200 строк** с хардкодом hex-офсетов

### Тесты
8. **Тесты несовместимы с pytest** — кастомные раннеры, `sys.stdout` corruption, 0 собираемых тестов

## 🟡 Крупные проблемы (18)

### Физика/Алгоритмы
9. **Несогласованность плоскости поля** — половина функций `k=sin(θ)` (X-Z), половина `l=sin(θ)` (Y-Z). Метки T/S переставлены
10. **Инвариант Зейделя неверен** — `Δ(nu)` вместо `n·(u+h/R)`. Абсолютные и относительные вклады поверхностей ошибочны
11. **Лучи не наводятся на входной зрачок** — старт с z=-50, игнор sP. Для 5° поля смещение ~4.4 мм
12. **OPL пересчитывается неверно** в волновых функциях — игнорирует `result.opl`, нет опорной сферы
13. **Конечный конъюгат: z=-50 хардкод** — `/50` в знаменателе для любого расстояния объекта
14. **Виньетирование использует параксиальное преломление** вместо реального трассировки
15. **HOLOGRAM/GRATING/TOROIDAL определены, но не трассируются** — молча считаются сферическими

### Качество кода
16. **Мутабельный хак semi_diameter** — `×1.4` перед вычислением, восстановление после. При исключении система повреждена
17. **Bare `except:`** в декодере и glass_catalog — глотает все ошибки
18. **Магические числа** — `1e-10`, `1e-15`, `1e-12`, `1e6` без именованных констант
18. **Мёртвый код** — мёртвые вычисления, закомментированные блоки, `val_70` читается но не используется
19. **Функции-гиганты** — `trace_ray_through_system` (75+ строк), `paintEvent` (180+ строк), `compute_field_aberrations` (150+ строк)
20. **Дублирование "bogus aperture"** — одна и та же эвристика 6+ раз
21. **Смешение RU/EN** — `'ЗЕРКАЛО'`, `'ВОЗДУХ'` как логические маркеры
22. **`compute_all_analysis()` — мёртвый код** — дублирует `_do_calc_phase2`, не используется

### Тесты/CI
23. **7 модулей с нулевым покрытием** — zernike (328 строк), visualization3d (501), opj_writer (93), glass_agf (196), optics_utils (106), map_images (48), worker (31)
24. **6 мёртвых тестов** — `test_mtf.py`, `test_achromat.py` и др. просто печатают, без assert
25. **`qa_v5_check.py` (272 строки)** — тестирует несуществующий API, мёртвый код
26. **46 скриптов в `scripts/`** — одноразовые отладочные скрипты замусоривают репо

## 🚀 Практики для ускорения разработки

### Разработка фичей
1. **Feature branches** — `feature/zmx-export`, `fix/aspheric-derivative`
2. **Маленькие коммиты** — один логический кусок = один коммит
3. **Regression test на каждый баг** — правило уже работает, расширить
4. **Code review** перед merge — даже self-review через `git diff`

### Интеграция фичей
- **Strategy pattern для новых типов анализа** — каждый виджет реализует `set_data(system)` + `get_table_data()` → добавление = новый файл + регистрация
- **Pipeline pattern для вычислений** — `AnalysisPipeline.add_task(name, fn)` вместо хардкода 16 задач
- **Dataclass schema для результатов** — `@dataclass AnalysisResults` с типами вместо dict с string keys

### Качество кода
- **Type hints везде** — `mypy --strict` как цель
- **Named constants** для всех магических чисел
- **English variable names** — русские только в строках/UI
- **Docstrings** на все публичные функции (Google style)

### CI/CD
- **pre-commit hooks** — black, isort, flake8
- **pytest на каждый push** — GitHub Actions
- **Coverage gate** — не ниже 60% для новых коммитов
